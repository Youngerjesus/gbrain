# Harness Contract

Use this reference before implementing shared loop state, adapter interfaces, trace records, actor roles, reuse paths, or resume behavior.

## Shared Harness Must Own

- loop family selection result and lifecycle;
- iteration ids, budgets, retry limits, stop conditions, and fail-fast checks;
- state transitions such as `PASS`, `FAIL`, `ACCEPT`, `RETRY`, `REJECT`, `BLOCKED`, and `FAIL_FAST`;
- generator, evaluator, judge, reviewer, and adjudicator call contracts;
- evaluation record envelope and decision envelope;
- trace and running-log capture;
- artifact inspection hook registration;
- stale artifact and stale feedback rejection;
- resume/reopen handling, including last meaningful update and partial trace recovery;
- Contract Gauntlet lifecycle, stage recording, freshness checks, fail-closed behavior, and downstream-block reporting for generated-output boundaries.

## Shared Harness Must Not Own

- domain-specific success criteria;
- domain thresholds or rubrics;
- artifact meaning, business quality, safety semantics, or domain truth;
- severity classification rules beyond requiring structured severity fields;
- scoring scripts or judge prompts that only make sense for one domain;
- search/prune criteria for a particular problem space;
- product artifact identity copied from scheduler, process, session, or control-plane ids;
- domain-specific downstream schemas, tool names, or artifact identity rules that belong to an adapter.

## Domain Adapters Must Own

- task goal and artifact target;
- generator implementation or prompt binding;
- evaluator implementation, scoring scripts, rubrics, deterministic checks, and artifact inspection methods;
- severity classification as `critical`, `major`, or `minor`;
- accept/reject thresholds and evidence sufficiency rules;
- prune criteria for exploration;
- hypothesis model for diagnosis;
- real-domain pilot fixtures when the project has a safe local verification path;
- runtime dependency declarations, auth/cache requirements, and fallback/block behavior for Codex, subagent, MCP, plugin, browser, provider, subprocess, worker, or other external runtime dependencies;
- downstream artifact contract records, including producer, consumer, artifact identity source, expected schema or format, evidence/provenance ownership, failure classification, and verification method.

## Contract Gauntlet

Use a Contract Gauntlet before expensive, provider-dependent, multi-agent, renderer/deployer/tool, or full live execution when objective handoff facts can be checked earlier.

For each stage where generated output crosses into a downstream consumer, evaluator, runner, reviewer, renderer, deployer, scheduler, or external tool, record:

- stage id and owner;
- artifact producer and downstream consumer;
- candidate artifact identity and freshness source;
- expected schema, format, manifest, bundle graph, ordering, or cross-artifact reference rule;
- deterministic parser, schema, fixture, smoke command, renderer check, or structured review used for preflight;
- evidence/provenance owner and fields the generator is forbidden to invent;
- runtime dependency allowlist snapshot used by that stage;
- pass, block, repair, rerun, reopen, no-op, reject-duplicate, or escalation action;
- whether full live execution is allowed, blocked, or only weakly evidenced.

Candidate artifacts are untrusted until their downstream artifact contract preflight passes. A field name that looks valid is not enough when refs, provenance, cache, or artifact identity are stale or generated by the wrong actor.

When a generated artifact is a bundle or graph, preflight the manifest, identity mapping, ordering, and cross-artifact references, not only each file in isolation. If a new downstream consumer is discovered after generation, branch selection, search/prune, or artifact shape changes, extend or rerun the gauntlet before full live execution.

## Runtime Dependency Allowlist

Required MCP servers, plugins, browser tools, providers, subagents, subprocesses, caches, and auth material are allowed when the adapter explicitly declares them. The unsafe default is ambient/global inheritance.

Each declared dependency must include:

- owner stage;
- purpose;
- run-local environment or config source;
- auth, cache, or credential requirement;
- expected availability signal;
- failure classification such as `runtime_dependency_blocked`, `provider_blocked`, `quota_blocked`, or `credential_blocked`;
- fallback, retry, block, or human-escalation behavior.

Subprocesses, child agents, worker processes, and nested loop invocations must receive the intended run-local runtime environment unless the adapter records an explicit exemption. If a child succeeds only by silently inheriting global MCP, browser, provider, auth, or cache state, do not count that as a clean preflight pass.

## Downstream Artifact Contract Preflight

For every generated artifact that another stage consumes, define an adapter-owned preflight contract:

- artifact producer;
- downstream consumer;
- artifact identity and freshness source;
- expected schema, format, or inspectable shape;
- evidence and provenance source of truth;
- fields the generator, evaluator, or judge may only reference, never create;
- failure classification;
- verification method;
- retry, quarantine, no-promotion, or block behavior.

LLM-generated provenance, evidence ids, hashes, pass/fail states, and artifact identity must be rejected unless the adapter defines a safe system-owned source of truth for those fields.

## Actor Authority

| Actor | Authority | Must not do |
| --- | --- | --- |
| Generator | Create or revise candidate artifacts from task input and selected feedback. | Decide final pass, accept, reject, or hide evaluator feedback. |
| Evaluator | Produce structured scores, findings, regressions, artifact observations, evidence refs, and preferably one primary finding or next-focus item. | Mutate the artifact or make retry economics decisions alone. |
| Judge | Decide `ACCEPT`, `RETRY`, or `REJECT` for uncertain attempts using evaluator evidence and budget/risk rules. | Replace deterministic gates where objective checks exist. |
| Reviewer | Find defects, false positives, false negatives, missing tests, and risks from an independent perspective. | Become authoritative without adjudication. |
| Adjudicator | Reconcile multi-review findings, filter false positives, calibrate severity, and select actionable findings. | Treat all reviewer output as true or optimize for agreement count. |
| Harness | Enforce lifecycle, trace, budgets, state transitions, stale evidence checks, and reporting. | Encode domain judgment that belongs in an adapter. |

## Structured Records

Use the target repo's native data tools when available: typed objects, schemas, dataclasses, structs, JSON schema, validators, parsers, or typed event records.

Minimum iteration record:

- `run_id`
- `iteration_id`
- `loop_family`
- `state_before`
- `task_ref`
- `input_artifact_ref`
- `generator_output_ref`
- `evaluator_input_ref`
- `feedback_ids`
- `evaluation_result_ref`
- `primary_feedback_id` or `primary_finding`, when findings exist and one item can be responsibly prioritized
- `targeted_improvement`
- `regressions_or_worsening`
- `decision`
- `decision_reason`
- `next_action`
- `unresolved_risks`

State, severity, pass/fail, accept/retry/reject, and evidence quality decisions must not be inferred from substring presence. If the project only has text output, add a parser, schema, or explicit human/judge escalation before treating the output as authoritative.

## Trace Chain

The 1.5 validation chain must be visible:

1. first generator output reference;
2. evaluator feedback reference with stable feedback ids or categories;
3. second generator action reference that names the consumed feedback;
4. changed artifact or diff reference;
5. next eval result or stop/retry decision.

If the second generator action is similar to the first output and cannot be linked to feedback consumption, the harness is not verified.

## Running Log

Every loop report should include:

- current best score or best state;
- what changed this round;
- what improved;
- what worsened;
- next thing to try;
- remaining risks.

For score-based loops, separate proxy score movement from real-objective risk. For finding-based loops, separate closed findings from new or worsened findings.

Finding-based loops should keep the next generator action focused. Multiple findings can be retained for traceability, but the evaluation or adjudication record should prefer one primary finding when prioritization is possible.

## Existing Harness And Multi-Root Targets

Before creating code, inspect the target repo for existing loop, eval, prompt, retry, review, or trace utilities.

- Reuse when the existing utility already owns lifecycle mechanics cleanly.
- Extend when the missing piece is a narrow adapter, trace field, or validation hook.
- Replace only when the existing path cannot meet convergence, trace, or structured-state obligations.
- Defer when the target repo boundary, language, test command, or artifact owner is unclear.

For multi-root or polyglot repos, identify the artifact owner, native implementation language, test runner, eval runner, and writable trace/report location for the specific target. Do not assume one language or root for every loop.

## Resume And Re-Entry

A resumed or restarted loop must preserve:

- last meaningful update time;
- current best state or score;
- partial trace and running log;
- latest artifact ref and feedback ref;
- retry/restart count;
- reason for continuing, accepting, rejecting, or blocking.

Use fresh artifact refs after restart unless the previous artifact identity is proven current. Do not bind new feedback to stale artifacts.

Produced harnesses must also handle re-entry explicitly:

- detect a duplicate active run before starting a new active loop for the same artifact or task;
- detect terminal or already-done runs and refuse to replay them as in-progress work unless a reopen reason is recorded;
- decide `resume`, `restart`, `reopen`, `no-op`, `reject duplicate`, or `block` from structured run state;
- write trace and running-log records with run-scoped files, append-only events, locks, atomic writes, or another collision-safe mechanism native to the repo;
- mark stale active runs separately from failed runs and record cleanup, escalation, or resume action;
- preserve prior trace records rather than overwriting them during retry, restart, or interrupted sessions.

Contract Gauntlet re-entry follows the same rule. A prior preflight pass can be reused only when the current artifact identity, downstream consumer contract, runtime dependency allowlist snapshot, evaluator schema, and adapter version still match. Otherwise rerun, reopen, block, or record a no-op from structured state.

Failed or partial preflight artifacts may remain available for diagnosis, but they must be marked as not promotable and not consumable by downstream stages. If one gauntlet stage passes and a later stage fails, record the failed stage, invalidated downstream stages, protected consumer, retry/resume action, and operator-facing failure reason.
